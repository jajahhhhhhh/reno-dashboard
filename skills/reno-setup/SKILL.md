# Reno Setup

Run this skill once when the user first installs the plugin, or when they want to reconfigure (different sites, new contractor, moved their dashboard files). All other skills in this plugin read `CONFIG.md` from the working directory — without it, they fall back to generic placeholders and the experience degrades.

## When to run

- The user explicitly asks to set up, configure, or onboard.
- Another skill reports it cannot find `CONFIG.md` in the working directory.
- The user changes a fundamental fact (e.g., "we sold the Chaweng site", "switched contractors").

## Steps

### 1. Check for an existing CONFIG.md

Read `./CONFIG.md` in the working directory. If it exists, summarize the current values in plain language and ask if the user wants to (a) keep as-is, (b) edit one or two fields, or (c) start fresh. Skip to step 4 unless they want to start fresh.

### 2. Confirm dashboard files location

The plugin needs to know where the HTML dashboard files live. Ask:

> "Where are your dashboard HTML files? (the folder with `dashboard-final.html` and `inventory-system.html`)"

If the user is unsure, look in common locations: working directory, `./files/`, `./dashboard/`, `./reno-dashboard/`, or `~/Desktop/`. Use Bash `find` from the working directory to look:

```bash
find . -maxdepth 4 -name "dashboard-final.html" 2>/dev/null | head -5
```

Confirm the absolute path with the user before saving.

### 3. Gather configuration via AskUserQuestion

Ask the user **all of these in a single `AskUserQuestion` call** (don't drip-feed):

- **Sites** — "How many renovation sites are you tracking?" (1, 2, 3+)
- **Currency** — "What currency do you track payments in?" (THB, USD, EUR, Other)
- **Language** — "What language for reports and summaries?" (Thai, English, Bilingual)

Then ask for the names individually (these need free text, not multiple choice):
- Name of each site (e.g., "Lipa Noi", "Chaweng")
- Primary contractor name (e.g., "MR.HOME")

If the user already mentioned these facts in the current conversation, infer them and confirm rather than re-asking.

### 4. Write CONFIG.md

Write to `./CONFIG.md` in the working directory:

```markdown
# Reno Dashboard Configuration

> Generated by `reno-setup` on {YYYY-MM-DD}. Edit by hand or re-run the skill.

## Sites

- **~~site-a**: {Site A Name}
- **~~site-b**: {Site B Name, or "(single-site mode)"}
- **~~site-c**: {Site C Name, or omit}

## Contractor

- **~~contractor**: {Contractor Name}

## Money

- **~~currency**: {THB}
- **~~currency-symbol**: {฿}

## Output

- **~~language**: {Thai}

## Dashboard files

- **~~dashboard-path**: {/absolute/path/to/folder}
- `dashboard-final.html` — tasks + payments
- `inventory-system.html` — stock + transactions
- `index.html` — landing page

## Notes

{any custom notes the user wants future skills to know — e.g., "MR.HOME paid per-draw, not monthly", "Lipa site is owner-occupied, Chaweng is Airbnb"}
```

### 5. Validate

Run these checks and report results:

- Does `~~dashboard-path/dashboard-final.html` exist? (Use `Bash test -f`)
- Does `~~dashboard-path/inventory-system.html` exist?
- Does the dashboard HTML contain a `SAMPLE_TASKS` array? (`grep -c "SAMPLE_TASKS"`)

If any check fails, tell the user and offer to either correct the path or proceed anyway (some skills will degrade gracefully).

### 6. Confirm and suggest next step

Tell the user what was saved (in their chosen output language) and suggest a next skill to try:

- If they have lots of LINE chat to import → `reno-ingest-chat`
- If they want to see what's there → `reno-status`
- If they want to log a payment they just made → `reno-log-money`

## What NOT to do

- Don't ask about MCP servers or auth — this plugin has none.
- Don't try to set up GitHub Pages or any deployment. That's a separate concern.
- Don't ask about contractors plural unless the user volunteers there are multiple. Default to one.
- Don't fail loudly if CONFIG.md can't be written (e.g., read-only working directory). Save to `~/.reno-dashboard-config.md` as fallback and tell the user.
